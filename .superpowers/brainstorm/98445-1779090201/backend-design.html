设计三：后端 API 与数据流

FastAPI 如何封装 TradingAgents 并通过 SSE 推送实时更新

API 端点设计
POST /api/analyze
启动分析任务，立即返回 job_id
请求体: ticker, date, analysts[], language
响应: {"job_id": "uuid"}
GET /api/stream/{job_id}
SSE 流：实时推送智能体状态和报告片段
Content-Type: text/event-stream
推送直到分析完成或出错
GET /api/report/{job_id}
获取完整最终报告（Markdown 格式）
用于页面刷新后恢复内容 / 下载报告
SSE 事件格式
agent_status — 智能体状态变更
event: agent_status
data: {
  "agent": "新闻分析师",
  "status": "in_progress"
  // pending | in_progress | completed
}
report_section — 报告内容就绪
event: report_section
data: {
  "section": "news_report",
  "content": "## 新闻分析\n..."
  // Markdown 格式全文
}
final_decision — 最终交易决策
event: final_decision
data: {
  "action": "BUY",
  "quantity": "20%",
  "stop_loss": "$395",
  "take_profit": "$460",
  "rationale": "..."
}
done / error
event: done
data: {"job_id": "uuid"}

event: error
data: {
  "message": "分析失败原因"
}
TradingAgents → FastAPI 集成方式
现有 CLI 通过轮询 LangGraph stream chunks 检测智能体状态变化（cli/main.py MessageBuffer），我们复用同一机制：
①
POST /api/analyze 收到请求后，用 asyncio.create_task 在后台线程运行 TradingAgentsGraph.propagate()
②
复用 MessageBuffer 逻辑监听 LangGraph stream chunks，检测 agent_status 和 report_sections 变化
③
每次状态变化写入 asyncio.Queue，SSE handler 消费队列并推送给浏览器
④
分析完成后解析 final_trade_decision 字段，推送 final_decision 事件，然后推送 done
并发限制
每次只允许 1 个分析任务运行（避免 Mac Mini 8GB 内存压力）。如果已有任务在跑，新请求返回 HTTP 429，前端显示"当前有任务运行中，请稍后"。
